# X-Change AI業務管理システム

**吉田自動車工業 X-Changeタイヤ販売店舗向け プロトタイプ**

> 補助金申請前 説明資料 兼 プロトタイプ仕様書 (v3.18.10 / 仕入合計 集計式の店舗名条件を緩和 (案B) 修正版)
> **店舗側はスマートフォン操作 / 本社側はPC操作を想定。スマホでも本社側の確認が可能。**

> **経費レシートOCRは Tesseract.js によるブラウザ内処理のためAPIキーは不要、画像は外部送信されません（v3.17）。**

## 2. 現在の業務課題

吉田自動車工業様の X-Change タイヤ販売店舗の運営において、従来の業務には以下のような課題があります。

- **拠点分散**: タイヤ販売店舗は本社とは別拠点で運営されている
- **少人数運営**: 店長1名で店舗運営を行っている
- **手作業中心**: 売上報告が紙伝票・Excel での手作業になりやすい
- **レシート管理の遅延**: 経費レシートの提出や確認が遅れやすい
- **本社確認の遅延**: 本社側で売上・経費の確認に時間がかかる
- **月次決算の遅れ**: 月次決算や利益把握が翌月以降にずれ込む
- **利益の見えにくさ**: タイヤ販売部門単体の利益がリアルタイムに見えにくい

これらの課題を、AI と OCR を活用した業務管理システムで解決します。

## 3. 導入後の業務フロー

### 3-1. 売上フロー

```
[店舗] 1. 店長が売上内容を 音声 または テキスト で入力
        ↓
       2. AI が売上項目（商品名 / サイズ / 数量 / 単価 / 合計 / 支払方法 / 車種等）に自動分解
        ↓
       3. 店長が画面で内容確認・必要に応じて修正
        ↓
       4. 「この内容で登録」ボタンで売上登録
        ↓
[本社] 5. 売上一覧に 即時反映（未確認）
        ↓
       6. 本社事務が内容確認 → 「確認済み」 へ
        ↓
       7. 内容に問題があれば 「修正依頼」 をコメント付きで送信
        ↓
[店舗] 8. 修正依頼を受信 → 内容を修正して 「再提出」 → 再び未確認へ
```

### 3-2. 経費フロー

```
[店舗] 1. 店長が レシート画像 をアップロード（撮影 or サンプル選択）
        ↓
       2. OCR がレシートから 日付 / 購入先 / 金額 / 消費税 / 支払方法 / 内容 を読取
        ↓
       3. AI が 12 カテゴリから 経費科目を候補分類（Top3 + 全候補）
        ↓
       4. 店長が内容確認・必要に応じて修正
        ↓
       5. 「この内容で登録」ボタンで経費登録
        ↓
[本社] 6. 経費一覧に 即時反映（未確認、AI分類科目のみ確定済み）
        ↓
       7. 本社事務が 「本社確定科目」 を最終決定（必要なら変更）
        ↓
       8. 「確認済み」 へ
        ↓
       9. 内容に問題があれば 「修正依頼」 をコメント付きで送信
```

### 3-3. 月次フロー

```
[本社] 1. 月次集計画面で売上・経費を確認
        ↓
       2. 未確認・修正依頼中のデータを処理（確認済み / 差し戻し）
        ↓
       3. 月次集計（売上合計・経費合計・粗利益・件数・内訳）を確認
        ↓
       4. 売上CSV / 経費CSV / 月次集計CSV を出力
            ↓ （会計ソフト連携・補助金申請資料に利用）
       5. 「月次締め」 ボタンでその月のデータを 「月次処理済み」 へロック
```

#### 領域間の依存関係

| 依存方向 | 内容 |
|---|---|
| A → 共通 | 売上レコード（`records[].type === "sale"`）は localStorage `xchange.records.v1` に書き込まれ、領域C の本社画面で参照される |
| B → 共通 | 経費レコード（`records[].type === "expense"`）は同じく localStorage に書き込まれ、領域C の本社画面・月次集計・CSV出力で参照される |
| C → A・B | 本社が「修正依頼」を発行すると、対象レコードの `status` が `"修正依頼"` になり、領域A（売上）/領域B（経費）の修正再提出フローで参照される |

データのスキーマ（売上 / 経費レコードの項目）は §6 データ構造 を**領域横断の契約**として扱います。スキーマ変更を伴う修正は、3領域すべての影響範囲を確認してから行います。

#### 修正時のチェックリスト

- ✅ **領域A の修正** → 領域B（経費OCR）と領域C（本社確認・CSV列構成）を破壊していないか確認
- ✅ **領域B の修正** → 領域A（売上音声入力）の `parseVoiceText` / `parseStructuredFields` に手を加えていないか、本社の経費一覧・月次集計の表示が崩れていないか確認

- ✅ **共通スキーマ変更** → §6 データ構造の更新 + `migrateRecord()` への移行ロジック追加 + 領域A・B・C 全画面でのレンダリング確認

### 10-1. Google スプレッドシート連携（v3.17.4 新規 / v3.17.5 拡張）

> 📡 **売上・経費・仕入の3種類に対応**
> 「確認して登録」を押すと、既存の localStorage 保存に加え、Google Apps Script Web アプリ経由で Google スプレッドシートにも 1行追記されます。送信に失敗した場合でも、**ローカル保存は維持され、`sheetsSyncStatus` に `failed` が記録されます**。

| action | 送信先シート | 呼び出し元 | ステータス |
|---|---|---|---|
| `addSale`     | **01_売上明細** | 売上 登録前確認 →「確認して登録」 | ✅ 稼働中（v3.17.4） |
| `addExpense`  | **02_経費明細** | 経費 登録前確認 →「この内容で登録」 | ✅ 稼働中（v3.17.5） |
| `addPurchase` | **03_仕入明細** | 仕入 登録前確認（**専用画面未実装**） | 🟡 ヘルパ準備済 / 将来有効化 |

> 💡 **仕入登録画面について**
> 現プロトタイプには専用の仕入登録画面はなく、タイヤ仕入は「経費レシート登録」フローで `category: "タイヤ仕入"` として記録し **02_経費明細** へ送信されます。`sendPurchaseToSheets()` ヘルパ関数は実装済みで、将来 `type:"purchase"` 専用画面（商品名・タイヤサイズ・数量・単価・請求書番号・支払予定日・支払ステータス入力欄）を追加した際にそのまま呼び出せます。

#### 設定

`app.js` 冒頭の `XCHANGE_SHEETS_CONFIG` で制御します。

```javascript
const XCHANGE_SHEETS_CONFIG = {
  enabled: true,
  endpoint: 'https://script.google.com/macros/s/.../exec',
  token:    'XCHANGE_TOKEN_2026'
};
```

- `enabled: false` で送信を無効化（ローカル保存のみで動作）
- `endpoint` は Apps Script ウェブアプリの公開URL（アクセスは「全員」「自分として実行」推奨）
- `token` は Apps Script 側 `doPost(e)` で照合する共有秘密

#### 経費金額の優先順位（v3.17.6 確立）

> ⚠ **最終確認画面の入力欄が常に最優先**
> 経費の `amount` は、localStorage 保存・Google Sheets 送信のいずれも **`finalExpenseRecord.amount`** （登録ボタン押下時点の `#confExpenseAmount` 入力欄の値）を参照します。OCR推定値・AI分類・サンプル経費・直前の `draftExpense.amount` で上書きされません。

1. **最終確認画面の金額入力欄** に現在入っている値（`#confExpenseAmount`）← 常にこれが採用される

### 10-2. CSV出力のスマホ対応（v3.18.3 新規）

> 📌 **CSV出力はPCでは通常のダウンロード、スマホではブラウザの制限に備えて、保存リンク・共有・コピー・CSVプレビューを表示します。スマホで自動保存されない場合は、CSVを共有またはコピーして利用できます。**

PC ブラウザでは `<a download>` の自動クリックでダウンロードフォルダに保存できますが、スマートフォン (iOS Safari / Android Chrome 等) では自動ダウンロードが制限される場合があります。本プロトタイプでは **CSV出力モーダル** で 4 つの手段を提示し、どの環境でも確実にCSVを取り出せます。

#### モーダルの構成

| 要素 | 動作 |
|---|---|
| 📄 ファイル名 | 例: `xchange_sales_2026-05.csv` |
| 端末別メッセージ | **PC**: 「CSVをダウンロードしました。ダウンロードフォルダを確認してください。」<br>**スマホ**: 「CSVを作成しました。保存されない場合は『📥 CSVを保存』『📤 CSVを共有』『📋 CSVをコピー』を使用してください。」|
| 📥 **CSVを保存** | Blob URL を `href` に持つ `<a download>` リンク。**ユーザーがタップ** することで Safari/Chrome のダウンロードまたは「ファイル」アプリへの保存ダイアログが確実に起動。 |
| 📤 **CSVを共有** | `navigator.share({ files: [csvFile] })` で AirDrop / メッセージ / メール / Drive 等へ共有。File共有が不可な環境ではテキスト共有にフォールバック。Web Share 未対応ブラウザでは非表示。 |
| 📋 **CSVをコピー** | `navigator.clipboard.writeText(csv)` でクリップボードへ。失敗時はプレビュー textarea を自動 focus + select し「下のCSVプレビューを選択してコピーしてください」と案内。 |
| CSVプレビュー | 読み取り専用 textarea (10行)。長押し選択 → メモ / メール / Googleスプレッドシート に貼り付け可。最後の手段としても機能。 |

#### 技術仕様

```javascript
// downloadCSV(filename, rows) の主要処理
const bom = "﻿";
const csv = bom + rows.map(r => r.map(escapeCsvCell).join(",")).join("\r\n");
const blob = new Blob([csv], { type: "text/csv;charset=utf-8;" });
const url = URL.createObjectURL(blob);

// 1. PC 用の自動ダウンロード (スマホでは失敗する可能性あり、無視)
try {
  const a = document.createElement("a");
  a.href = url; a.download = filename;
  a.style.display = "none";
  document.body.appendChild(a); a.click(); a.remove();
} catch (_) {}

// 2. 保存リンク / 共有 / コピー / プレビュー モーダルを必ず開く
openCsvExportModal({ filename, csv, blob, url });
// URL は close 5秒後に revokeObjectURL (保存リンクのタップ猶予を確保)
```

X-Change アプリ (Web版) からの売上/経費/仕入データはすでに Google スプレッドシートに送信されています（addSale / addExpense / addPurchase）。第4分割では、**スプレッドシート側の自動計算式と補助シートを Apps Script で一括セットアップ** できます。

#### 提供ファイル

```
apps-script/X-Change-Aggregation.gs
```

このファイルは X-Change アプリ本体とは独立した **Google Apps Script** 用のソースです。スプレッドシート側の Apps Script エディタに追加して使用します。**既存の `doPost` / `addSale` / `addExpense` / `addPurchase` には一切手を加えません。**

#### セットアップ手順

1. スプレッドシートを開く → 「拡張機能」→「Apps Script」
2. 左ペインの「+」→ 「スクリプト」で **新しい .gs ファイル** を作成（既存ファイルは上書きしない）
   - ファイル名: `X-Change-Aggregation`
3. このリポジトリの [apps-script/X-Change-Aggregation.gs](apps-script/X-Change-Aggregation.gs) の内容を全てコピーして貼り付け
4. 関数選択メニューから `setupMonthlyAndClosingFormulas` を選択 → 実行（▶️）
5. 初回は OAuth 認可ダイアログが出るので **許可**
6. スプレッドシートを再読込 → 上部メニューに **「X-Change」** が表示される

#### 【超軽量】まず実行する 2 関数（v3.18.4.3 LITE・最優先）

> ⚠ **重い場合は使わず、`setupXChangeMinimumSheets` → `setupXChangeBasicFormulasOnly` の順に実行してください。**
>
> 全シート全行走査・全データ正規化・条件付き書式・複雑な QUERY 式は一切実行しません。各関数は **10〜30 秒以内** で確実に完了します。

| 順番 | 関数 | 内容 | 想定実行時間 |
|---|---|---|---|
| ① | **`setupXChangeMinimumSheets`** | 05_月次集計 / 06_決算用集計 / 05A〜06B 補助シートの **作成とヘッダーだけ**。計算式・書式は設定しない | <10 秒 |
| ② | **`setupXChangeBasicFormulasOnly`** | 05_月次集計 と 06_決算用集計 に **最小限の計算式だけ** (売上/経費/仕入/粗利益/未確認/修正依頼)。バッチ `setFormulas` で 1 API 呼出/列 | <30 秒 |

**LITE 版の特徴**:

- **per-cell `setFormula` を使わない** → 各列の式を配列に蓄積し、`setFormulas` で 1 API 呼出/列にまとめる
- **ヘッダー検索を行わない** → X-Change アプリ送信時の既定列位置 (A=日付/B=店舗名/K=売上合計 等) をハードコード
- **getLastRow を使わない** → 月次=2〜13 行固定 / 決算=2〜6 行固定
- **条件付き書式・列幅調整・複雑な QUERY/JOIN 式は設定しない** → 純粋な SUMIFS/COUNTIFS のみ
- **日付・金額の正規化は実行しない** → 必要なら別途 `normalizeXChangeDatesOnly` / `repairXChangeAmountsOnly` を手動実行

**LITE で設定される計算式の範囲**:

| シート | 列 | 内容 |
|---|---|---|
| 05_月次集計 | C | 売上合計 (= SUMIFS over saleDate × storeName 一致) |
| 05_月次集計 | D | 売上件数 (= COUNTIFS) |
| 05_月次集計 | E | 仕入合計 |
| 05_月次集計 | F | 経費合計 |
| 05_月次集計 | G | 粗利益 (= C − E − F) |
| 05_月次集計 | N | 未確認件数 (売上+経費+仕入) |
| 05_月次集計 | O | 修正依頼件数 (修正依頼+差戻し × 3 type) |

| 06_決算用集計 | G | 年間経費合計 |
| 06_決算用集計 | P | 未確認データ件数 (未確認+修正依頼+差戻し+保留 × 3 type) |

#### セットアップを分割実行する（v3.18.4.2 — heavy 版）

> 📌 **月次・決算用集計セットアップは、Apps Scriptの実行時間制限を避けるため、ヘッダー作成、補助シート作成、月次計算式、決算計算式、表示形式、日付正規化、金額正規化を分割して実行できます。初回は setupXChangeHeadersOnly → setupXChangeAuxSheetsOnly → setupXChangeMonthlyFormulasOnly → setupXChangeYearEndFormulasOnly → setupXChangeFormatsOnly の順に実行してください。**

`setupMonthlyAndClosingFormulas()` が長時間 (5分以上) 停止する場合、以下の **分割関数を 1 つずつ実行** してください。各関数は 30 秒〜1 分以内で完了するように軽量化されています。

| 順番 | 関数 | 内容 |
|---|---|---|
| ① | **`setupXChangeHeadersOnly`** | 05_月次集計 / 06_決算用集計 のヘッダー設定 + 補助シート (空) 作成 |
| ② | **`setupXChangeAuxSheetsOnly`** | 05A / 05B / 05C / 05D / 06A / 06B 補助シートの整備 |
| ③ | **`setupXChangeMonthlyFormulasOnly`** | 05_月次集計 の計算式設定 (2〜13 行・12 ヶ月固定) |
| ④ | **`setupXChangeYearEndFormulasOnly`** | 06_決算用集計 の計算式設定 (2〜6 行固定) |
| ⑤ | **`setupXChangeFormatsOnly`** | 表示形式 / 列幅 / 1 行目固定 / 条件付き書式 |
| 任意 | `normalizeXChangeDatesOnly` | 日付列を `yyyy-mm-dd` に正規化（既存データが多いと重くなる可能性。**自動実行しない**） |
| 任意 | `repairXChangeAmountsOnly` | 金額列を数値化（同上。**自動実行しない**） |

**`setupMonthlyAndClosingFormulas()` (一括) は引き続き利用可能** — 内部で①〜⑤を順に呼び出します。データ量が少なく、Apps Script の 6 分制限内で完了する場合はそのまま使えます。

**タイムアウト対策の主な改修内容**:

- 全 setup 関数で **固定行数化** (`Math.max(sh.getLastRow(), ...)` → 月次=12行 / 決算=5行 固定)。手入力や空白で `getLastRow()` が巨大値を返してもループが暴走しない
- 日付・金額正規化は **明示的な手動実行** のみ。自動セットアップフローには含めない
- スプレッドシート上部「X-Change」メニューに **分割関数を個別に表示** (①〜⑤の番号付き)

#### 「X-Change」メニュー

メインエントリは `agg_installMenu()` で、上部にカスタムメニューを追加します。**`onOpen` という名前は既存 `コード.gs` と衝突する可能性が高いため、本ファイルでは `agg_installMenu` に変更しています。** メニューを自動で表示する方法は 3 つから選択できます:

2. **方法B** — 既存 `コード.gs` に `onOpen` がある場合: その関数の末尾に `agg_installMenu();` を 1 行追加
3. **方法C** — 自動表示が不要な場合: 必要時に Apps Script エディタから `agg_installMenu` を選択 → 実行（1回だけ）

#### 05_月次集計 の計算

A列「対象年月」（例: `2026/05`）と B列「店舗名」を入れた行に対し、以下の列に SUMIFS / COUNTIFS 式が自動設定されます（2〜13行目の12ヶ月分）。

| 列 | 内容 | 計算式の要旨 |
|---|---|---|
| C | 売上合計 | `SUMIFS(01_売上明細!totalAmount, saleDate>=月初, saleDate<=月末, storeName=B)` |
| D | 売上件数 | `COUNTIFS(...同上)` |
| E | 仕入合計 | `SUMIFS(03_仕入明細!totalAmount, ...)` |

| O | 修正依頼中件数 | `confirmStatus="修正依頼" or "差戻し"` 合計 |

#### 列・ヘッダ自動検出

Apps Script は **ヘッダ名で列を検索** するため、列順が前後しても動作します（`colLetterByHeader_` 関数）。X-Change アプリの送信 payload キー（`saleDate` / `storeName` / `totalAmount` / `confirmStatus` 等）が 1 行目のヘッダになっていれば、Apps Script 側は列文字を自動取得して式を構築します。

| 03_仕入明細 | A=purchaseDate, B=storeName, I=totalAmount, J=paymentMethod, K=confirmStatus, L=paymentStatus, M=sheetsSyncStatus |
| 04_営業日管理 | A=日付, B=店舗名, C=営業区分 |

#### 日付・金額の正規化

送信されてくるデータは文字列形式（`"2026-05-19"`、`"3000"`）の場合があります。SUMIFS は **YYYY-MM-DD 文字列** であれば期間フィルタが正しく動作するため、特別な変換なしで集計可能です。

万一、別形式が混入した場合は **「X-Change」メニュー → 「日付列を正規化」/「金額列を数値化」** を実行することで:

- 日付: `2026/05/19` `2026.5.19` `2026年5月19日` → `2026-05-19` に統一
- 金額: `¥3,000` `3000円` `１５０００` → 数値 `3000` / `15000` に変換

#### 表示形式 (見た目)

- 1行目を固定 (`setFrozenRows(1)`)
- 金額列を `¥#,##0` 通貨表示
- 件数列を `#,##0` 整数表示
- 改行表示が必要な列 (支払方法別/売上区分別/商品別/経費科目別) は `setWrap(true)` で折返し
- ステータス列に **条件付き書式**:
  - 「未完了」「確認未完了」「差戻し」 → 赤系背景
  - 「修正依頼」 → 黄系背景
  - 「保留」 → 青系背景
  - 「締め可能」「確認可能」 → 緑系背景
  - 「締め済み」「確認済み」 → 灰系背景
- 列幅を内容に合わせて調整

#### Sheets 送信失敗 / 送信状態の確認

送信失敗データは X-Change アプリ側「Sheets同期状態」タブ（v3.18.2）で確認・再送信できます。スプレッドシート側でも各レコードの `sheetsSyncStatus` 列で確認可能です。月次集計・決算用集計の対象データには `synced` / `failed` / `pending` が混在しますが、計算式は全件を対象にします（失敗データも localStorage には登録されているので、再送信後に Sheets 側にも反映されます）。

#### 既存データを壊さないこと（セットアップ時の安全性）

- `setupMonthlyAndClosingFormulas` は **ヘッダ行が空欄の場合のみ** ヘッダを書き込みます。既存ヘッダは触りません
- 計算式は 2 行目以降の **5/6 シートの集計列に対してのみ** 書き込みます
- 01_売上明細 / 02_経費明細 / 03_仕入明細 / 04_営業日管理 / 07_店舗マスタ / 08_商品マスタ / 09_経費科目マスタ / 10_支払方法マスタ / 11_本社確認管理 / 12_操作ログ の **データ行 (2 行目以降)** には一切触りません
- ステータス列で「締め済み」「確認済み」が手動入力されている場合は **式で上書きしません**
- 既存 `doPost / addSale / addExpense / addPurchase` には影響なし

#### 経費登録 OCRモード・手入力モード（v3.17.8 新規）

> 📌 **経費登録は、OCR読取モードと手入力モードに対応します。OCRは補助機能であり、手入力モードではOCR結果やサンプル値を使わず、ユーザーが入力した金額・購入先・内容をそのまま登録します。レシート画像は証憑として添付できます。**

経費レシート登録画面の最初に、登録方法を2つから選びます。

| モード | 用途 | 入力方法 |
|---|---|---|
| **① レシートをOCRで読み取る** | レシート画像があり、OCR + AI分類で素早く登録したい | 画像アップロード → 実OCR → 確認画面で修正 → 登録 |
| **② 手入力で登録する** | OCRで読み取れないレシート、手書き、または直接入力したい | フォームに直接入力 → レシート画像は任意添付 → 確認画面で再確認 → 登録 |

**手入力モードの仕様（OCR推定値・サンプル値を一切使わない）:**

- 入力欄: 経費日 / 購入先 / 内容 / 金額（税込）/ 消費税 / 支払方法 / 経費科目 / レシート画像（任意）/ 備考
- 店舗名・担当者はシステム既定値（吉田自動車工業 X-Change本店 / 店長）。確認画面で編集可能
- 本社確定科目（`hqCategory`）は **本社が確認時に設定** するため店舗入力欄では未表示。送信時は経費科目 (`category`) からフォールバックされる
- 「次へ確認」を押すと OCR を一切実行せず、確認画面に直接遷移
- 確認画面では金額・購入先・内容・消費税・支払方法・経費科目・備考をすべて編集可
- 「この内容で登録」押下時、**確認画面の DOM 入力値** から `finalExpenseRecord` を組み立て、localStorage 保存と Google Sheets 送信の両方に **同一オブジェクト** を使用
- 手入力レコードは `ocrText: ""` / `ocrSource: "manual"` / `source: "manual"` / `receiptThumb: "✍️"` で記録。OCR読取結果カードは確認画面で **非表示**

**レシート画像の保存:**

手入力モードで添付した画像は、現在の仕様では `receiptDataUrl` (base64 dataURL) として localStorage に保持し、Google Sheets 02_経費明細 の `receiptImageUrl` カラムには同 dataURL をそのまま送信します。Apps Script 側で Drive へのアップロード・URL 化を行う **外部URL保存は次フェーズ** で実装予定です（現段階では base64 文字列が大きく Sheets セルの上限に達する場合は Apps Script 側で省略保存される可能性があります）。

**デバッグログ（DevTools コンソール）:**

[Expense] final expense record           {...}        // OCRモードで登録した場合
[Expense] final amount input value       "3000"       // 両モード共通

#### OCR結果の安全な扱い（v3.17.2 改善）

> ⚠ **OCR読取結果をそのまま登録項目へ流し込みません。**
> 推定に自信がない項目は **空欄または「要確認」** にし、店長による確認・修正を必須化します。OCR全文は「OCR読取結果（原文）」欄にのみ表示し、「内容」欄には短い要約だけが入ります。

| 項目 | 抽出ロジック | 自信がない時の挙動 |
|---|---|---|
| **金額（税込）** | ① `総合計` / `合計（税込）` / `税込合計` ② `お支払金額` / `請求金額` ③ `合計`（小計を除外、最後の出現を採用）④ `税込` ⑤ `クレジット支払` / `カード支払` / `VISA/Master/JCB/AMEX` ⑥ `現金` / `お預かり` / `お買上金額` ⑦ ¥/円マーク付き数字の最大値（≥50円のみ） | 0円のまま空欄。**「金額を確認してください」** で登録ブロック |
| **消費税** | `消費税等` / `税額` / `内消費税` / `消費税(8%)` / `消費税(10%)` キーワード直近の金額。`8`/`10` 単独はレートとして除外 | 高信頼な合計から自動逆算（10%税込前提）。それ以外は空欄 |
| **購入先** | レシート上部の店舗名（セブン-イレブン / 7&i / ローソン / ファミマ / ENEOS / カインズ / コメリ / オートバックス / イエローハット 等を網羅） | 空欄。**「購入先を入力してください」** で登録ブロック |
| **支払方法** | クレジット/VISA/Master/JCB/AMEX → クレジットカード ／ PayPay/d払い/楽天ペイ → QR決済 ／ 現金/お預り/お釣り → 現金 | 空欄。**「支払方法を選択してください」** で登録ブロック |
| **AI経費科目** | 購入先 + OCRテキストからルールベース分類 | 低信頼度時は **未選択** にして店長に手動選択を促す。**「経費科目を選択してください」** で登録ブロック |
| **内容** | 高信頼カテゴリがあれば「ガソリン代」「タイヤ仕入」等の短いラベル ／ 購入先のみなら「<購入先>購入分」 ／ どちらも不明 → **「レシート内容 要確認」** | OCR全文を「内容」欄に流し込まない |

**金額として安易に採用しない数字:** 電話番号 / 会員番号 / 伝票番号 / 承認番号 / 日付 / 時刻 / 商品コード / バーコード / ポイント / 税率（8 / 10）/ 数量 / 50円未満の小さな数字 / OCRノイズ。

**低信頼度バナー:** 金額が ¥/円マーク付きの最大値フォールバックでしか取れなかった場合、または購入先が判定できなかった場合、確認画面に **「⚠ OCRの読み取り精度が低いため、金額・購入先・経費科目を手入力してください。」** バナーを表示し、AI経費科目の自動選択をスキップします。

#### 実OCR読取とサンプルOCR読取の厳密分離（v3.17.1 改善）

> ⚠ 経費レシート登録では、**実OCR読取** と **サンプルOCR読取** を厳密に分けています。
>
> **実OCR読取を行った場合は、アップロードされたレシート画像から読み取った文字列のみを使って、購入先・金額・支払方法・AI経費分類候補を推定します。サンプル経費データは、サンプルボタンを押した場合のみ使用します。**
>
> アップロード画像があるのに OCR が空文字列を返した場合や、OCR読取に失敗した場合でも、**サンプル経費データで勝手に項目を埋めることはありません**（推定できなかった項目は空欄のまま登録確認画面に進みます）。

| シナリオ | 動作 |
|---|---|
| ✅ レシート画像をアップロード → 「OCR読取」 | Tesseract.js で実OCR実行。**サンプルデータは使用しない** |
| ✅ サンプルカードをタップ → 「OCR読取」 | サンプル固定OCRテキストを表示。実OCRは走らせない |
| ✅ 画像なし + メモ入力 → 「OCR読取」 | メモテキストをOCR結果として展開 |
| ✅ 画像なし + メモなし → 「OCR読取」 | 「先にレシート画像をアップロードしてください」を表示。**勝手にサンプルを使わない** |
| ✅ ユーザーがOCR結果を手動修正 → 「AI分類して確認」 | 修正後テキストを最優先で利用 |

画面上部には常に **OCR利用状況チップ**（未読取 / 実OCR結果を使用中 / サンプルOCR結果を使用中 / メモテキストを使用中）が表示され、現在どのソースで動作しているかが一目で確認できます。

開発確認用に画面下部とブラウザの DevTools コンソール (`console.debug("[v3.17 OCR]", ...)`) に **OCRソース・テキスト長・検出した購入先/金額/AI分類** が出力されます（本番表示では小さなグレー文字で目立たないようにしています）。

#### 利用フロー（店舗・スマホ）

1. 経費レシート登録画面を開く
2. 「📷 タップして撮影 / 画像を選択」 → スマホ背面カメラでレシートを撮影
3. 「🔍 OCR読取」をタップ
   - 初回のみ Tesseract.js 本体（約2MB）と日本語学習モデル（約15MB）を CDN からダウンロード（進捗バー表示）
   - 2回目以降はブラウザキャッシュから即起動
4. 認識結果が **編集可能テキストエリア** に表示される
5. 誤読箇所を直接修正 → 下のフィールドプレビュー（日付・金額・購入先・消費税・支払方法）が自動更新
6. 「🤖 AI分類して確認」 → AI経費分類候補（Top3）と共に登録確認画面へ

#### OCR精度に関する注意事項

OCR読取は以下の条件で精度が変わります。読み取り結果は **登録前に必ず確認・修正** してください。

- レシート画像の **明るさ・コントラスト**（暗い場所での撮影は推奨しません）
- **手ブレ・ピンぼけ**（撮影時に少し離れて、フォーカスを合わせてください）
- **文字の向き・歪み**（斜めから撮影すると認識率が下がります）
- **印字の薄さ**（感熱紙レシートで時間が経過したものは特に注意）
- **改行・桁区切り**（金額の `5,000` が `5.000` 等と誤読されることがあります）

そのため、画面上には常に **「読み取り結果は登録前に必ず確認・修正してください」** という注意書きを表示しています。

#### 本番化時のOCR強化（v3.17 → 本番）

> 📌 **本番運用上の注意**
>
> 現在のプロトタイプではTesseract.jsによるブラウザOCRを使用しています。ブラウザOCRはAPIキー不要でデモに適していますが、**日本語レシートの読取精度には限界があります**（手書き・薄い印字・斜め撮影・小さな文字サイズなどで誤認識が発生します）。本番運用では、**Google Vision OCR、Google Document AI、またはレシート専用OCR API をサーバー側で利用し、APIキーを安全に管理する構成**を推奨します。

現在のプロトタイプでは、**APIキーを使わないブラウザOCR（Tesseract.js）** によりレシート画像の文字読取を行います。これによりサーバー不要・追加コスト不要で実OCRデモが可能ですが、本番運用では以下の高精度OCRサービスへの切り替えを推奨します。

| 観点 | プロトタイプ（v3.17） | 本番推奨 |
|---|---|---|
| OCRエンジン | Tesseract.js（ブラウザ内） | **Google Cloud Vision OCR** または **Google Document AI**（Receipt Parser テンプレート） |
| 精度 | 印字レシートで実用レベル、手書き・薄い印字・斜め撮影は弱い | 手書き・低品質画像・縦書き等にも強い、レシート専用パーサあり |
| 構造化抽出 | 自前の正規表現で日付・金額・購入先を抽出 | 日付・店名・小計・税額・支払方法・明細行が **構造化JSONで返却** される |
| 通信 | 画像は外部送信されない | サーバーAPI経由（認証 + データ保護対応） |
| コスト | 無料（CDN通信のみ） | 1,000ページあたり $1.50（Vision OCR）／ Document AI は別料金体系 |
| 認証 | なし | API キー or サービスアカウント JSON、本番では Cloud Functions 経由でのプロキシを推奨 |

本番化時の実装は次の手順で行います:

1. Cloud Functions / Cloud Run に **OCR専用エンドポイント** を作成し、Vision API のキーをサーバー側に閉じ込める
2. フロントエンドからは画像 base64 を POST → サーバーで Vision OCR 呼出 → JSON で日付/購入先/金額/税額/支払方法/明細を返す
3. 既存の `runRealOCR()` 関数を **Cloud Functions 呼出** に差し替え（`parseExpenseText()` の正規表現抽出はフォールバックとして残す）
4. オフライン時 / API障害時のフォールバックとして **Tesseract.js を引き続き利用** することも可能（プログレッシブ・エンハンスメント）

#### 技術構成（v3.17）

```html
<!-- 遅延ロード: 経費レシート登録画面で「OCR読取」を初回タップしたタイミングで CDN から取得 -->
<script src="https://cdn.jsdelivr.net/npm/tesseract.js@5/dist/tesseract.min.js"></script>
```

```javascript
// 言語: 日本語 + 英語（金額の数字・英字混在に対応）
await Tesseract.recognize(dataUrl, "jpn+eng", {
  logger: (m) => updateOCRProgress(m.status, m.progress)

```

- **Tesseract.js v5** を採用（最新の WebAssembly 高速化版）
- 言語コード `"jpn+eng"` で日本語・英語の混在認識（金額・店名のローマ字混在対応）
- 進捗ロガーは Tesseract.js が出力する status 文字列を **日本語ラベルに変換** して表示
- 初回ダウンロード後は IndexedDB / Cache Storage に保存され、2回目以降は瞬時に起動

## 15. 補足: 会計処理 / 月次決算 / 補助金申請への活用

### 会計ソフト連携
- 売上CSV / 経費CSV をそのまま freee / マネーフォワード / 弥生 へ取り込み可能
- 本社確定科目が会計仕訳の科目と直結

### 月次決算
- 月次集計CSV 1 ファイルで PL（粗利益）と 内訳（科目別・支払方法別・商品別）を確認
- 月次締め処理でその月のデータを「月次処理済み」にロックし、改竄を防止

### 補助金申請資料
- 月次集計画面のスクリーンショット（5セクション）が、売上 / 経費 / 内訳 / 件数 をひと目で示す資料として利用可能
- 各画面の `.biz-note` 説明バナーが「この画面で何ができるか」を補助金申請担当者に明示

### 電子帳簿保存法
- レシート画像を `receiptDataUrl` として保存（本番では Google Drive 等への永続保管）
- 本番化時は タイムスタンプ・改竄検知 と組み合わせて電子帳簿保存法に対応可能
